Executive Summary

GBC OS is a well-designed internal operating platform built with vanilla HTML/CSS/JS and a Python data pipeline syncing from Square API. The UI is polished and the data model is rich. However, the system has critical security gaps that must be resolved before any client-facing deployment. There is no authentication, no authorization, no input sanitization, and sensitive business data (PII, financials, strategic notes) is fully exposed in client-side source code.

The architecture is a strong foundation for a white-label product, but requires a backend service layer, auth system, and data isolation before it can serve external clients.

Critical Severity

1.

No Authentication

CRITICAL

All portals are publicly accessible static HTML files. Anyone with the URL sees all partner data, invoices, contacts, financials, and strategic notes. No login, no session management, no access control.

2.

PII Exposed in Source Code

CRITICAL

Personal emails, phone numbers, business addresses, invoice amounts, and confidential notes embedded directly in HTML. Visible via View Source.

3.

Business Intelligence in Client Code

CRITICAL

Strategic notes contain competitive intel, agency warnings, pricing strategies, investor targets. Devastating if accessed by competitors.

High Severity

4.

XSS via innerHTML

HIGH

Multiple innerHTML assignments render data without sanitization. Functions emailLink() and phoneLink() return unescaped HTML.

5.

No Security Headers

HIGH

No CSP, X-Frame-Options, X-Content-Type-Options. Pages can be framed by malicious sites.

6.

Square API Token Risk

HIGH

Long-lived Bearer token with no rotation or scoped permissions.

7.

HubSpot IDs Hardcoded

HIGH

Portal ID and form GUIDs visible in source.

Architecture Assessment

Strengths

✓Clean SPA pattern with 15+ views
✓Rich data model (partners, contacts, locations, products, invoices, media, brand)
✓Automated Square API data pipeline
✓Drag-drop Kanban with persistence
✓Two quote calculators with HubSpot CRM integration
✓Polished dark UI with consistent design system

Weaknesses

✗Monolithic files (3,800+ lines single HTML)
✗All data embedded client-side
✗No component system or build tools
✗Duplicated logic between v2 and v3
✗Data sync only updates v2, not v3
✗No testing infrastructure
✗No error monitoring or analytics

Implementation Roadmap

Phase 1 — Security Foundation

Weeks 1–3
  • Authentication layer (Auth0 / Clerk / Supabase Auth)
  • Move sensitive data behind API
  • Role-based access control (Admin, Manager, Viewer)
  • CSP headers and security headers
  • Input sanitization on all rendering
  • Server-side validation for quote forms

Phase 2 — Architecture Modernization

Weeks 4–8
  • Backend API (FastAPI or Express)
  • Database migration (JSON to PostgreSQL)
  • Multi-tenant data isolation
  • Component-based frontend
  • Build tooling and bundling

Phase 3 — Product Features

Weeks 9–14
  • White-label theming engine
  • Client self-service onboarding
  • Webhook integrations (Stripe, QuickBooks, Toast POS)
  • Email notifications and alerts
  • Export/reporting (PDF, CSV)

Phase 4 — Scale and Launch

Weeks 15–20
  • Automated testing suite
  • CI/CD with staging environments
  • Error monitoring (Sentry) and analytics
  • Documentation and API docs
  • Pricing and billing integration
  • Beta client onboarding
